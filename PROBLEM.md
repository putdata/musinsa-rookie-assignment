# 대학교 수강신청 시스템

## 과제 개요

당신은 대학교 IT팀에 새로 합류한 개발자입니다.

기획팀에서 수강신청 시스템 구축을 요청했지만, 요구사항 문서가 충분히 명확하지 않습니다. 담당자에게 추가 질문을 할 수 없는 상황이며, 주어진 정보를 바탕으로 **합리적인 판단과 결정**을 내려야 합니다.

당신의 역할은 요구사항을 분석하고, 누락된 부분을 도출하며, 이를 문서화하고 구현하는 것입니다.

---

## 기획팀 요청사항

> **기획팀 메모**
>
> 매 학기 수강신청 기간마다 서버가 다운되어 학생들의 불만이 폭주하고 있습니다.
>
> 이번에는 제대로 된 시스템을 만들어주세요. **정원이 1명 남은 강좌에 100명이 동시에 신청해도, 정확히 1명만 성공해야 합니다.** 절대 정원을 초과해서는 안 됩니다.
>
> 아래는 필요한 기능 목록입니다:
>
> - 학생 목록 조회
> - 강좌 목록 조회 (전체, 학과별 - 정원/현재 인원/시간 정보 포함)
> - 교수 목록 조회
> - 수강신청
> - 수강취소
> - 내 시간표(이번 학기) 조회
>
> 참고로, 학생당 최대 **18학점**까지 신청 가능하고, 같은 시간대에 두 개 강좌를 들을 수는 없겠죠?
>
> 화면은 프론트팀에서 따로 작업할 예정이니, **서버 쪽만** 부탁드립니다.
> API 문서를 잘 만들어주시면 프론트팀에서 연동하기 편하겠죠?
>
> 잘 부탁드립니다.

## 기술 요구사항

### 기술 스택

| 항목 | 요구사항 |
|------|----------|
| 언어 | Java, Kotlin, Python, JavaScript 중 선택 |
| 프레임워크 | 자유 선택 (Spring Boot, FastAPI, Express 등) |
| 빌드 도구 | 언어에 맞는 표준 빌드 도구 사용 |
| 데이터베이스 | 인메모리 또는 실제 DB (자유 선택) |
| API 형식 | **REST API** |

### 헬스체크 엔드포인트

서버 정상 구동 확인을 위해 아래 엔드포인트는 **반드시** 구현해야 합니다:

```
GET /health
응답: HTTP 200 OK
```


### API 설계

- REST 원칙에 따라 API를 설계하세요
- 설계한 API는 문서화하여 `docs/` 디렉토리에 포함하세요

**API 문서 요구사항**:
- 모든 엔드포인트의 요청/응답 형식 명시
- 에러 케이스별 응답 정의 (HTTP 상태 코드, 에러 메시지)
- 형식 자유 (Markdown, OpenAPI 등)
- 평가자가 API 문서만으로 테스트를 작성할 수 있을 정도로 구체적이어야 함
- (권장) 요청/응답 예제를 포함하면 평가에 도움이 됩니다

> 참고: OpenAPI(Swagger) 형식은 API를 명확하게 정의하는 좋은 예시입니다.

### 동시성 제어

동시성 제어는 이 과제의 핵심 요구사항입니다.

- 정원 초과를 방지하는 동시성 제어 메커니즘을 반드시 구현하세요
- `docs/REQUIREMENTS.md`에 사용한 동시성 제어 전략을 설명하세요 (락 메커니즘, 트랜잭션 격리 레벨 등)
- 평가 시 동시 요청 시나리오로 직접 검증합니다

## 데이터 구축 요구사항

### 데이터 생성

서버 시작 시 테스트를 위한 초기 데이터가 준비되어야 합니다.
- `GET /health` → 200 응답 시점에 API 호출이 가능해야 함
- **1분 이내** 완료

### 데이터 규모

| 항목 | 최소 수량 |
|------|----------|
| 학과 | 10개 이상 |
| 강좌 | 500개 이상 |
| 학생 | 10,000명 이상 |
| 교수 | 100명 이상 |

### 데이터 제약

1. **동적 생성 필수**
   - 데이터는 애플리케이션 실행 시 프로그래밍 로직으로 생성되어야 합니다
   - 정적 파일(SQL, CSV) 및 대량의 리터럴 데이터는 사용할 수 없습니다
   - 참고: 이름, 학과명 등 **소규모 토큰 목록**은 코드에 포함 가능하나, **완성된 레코드 단위 데이터**는 런타임에 생성해야 합니다

2. **현실적인 데이터**
   - "User1", "Course1" 같은 의미 없는 데이터는 지양합니다
   - 실제 대학교에서 사용할 법한 현실적인 이름과 패턴을 사용하세요
   - 예: 학과명(컴퓨터공학과), 강좌명(자료구조), 학생 이름(김민준)

   **과도한 수준** (불필요):
   - 실제 대학 커리큘럼 데이터베이스 크롤링
   - 통계청 인구 데이터 기반 이름 생성

3. **속도 제약**
   - 서버 시작 후 **1분 이내**에 데이터 생성 및 헬스체크 응답 완료

4. **데이터 정합성**
   - 생성된 데이터는 비즈니스 규칙을 위반하지 않아야 합니다
   - 데이터 생성 시 발생할 수 있는 문제들을 고려하세요

### 데이터 확인 및 테스트 편의성

기획팀 요청사항에 포함된 **학생/강좌/교수 목록 조회 API**를 통해 평가자가 유효한 ID를 확인할 수 있어야 합니다.

**강좌 목록 조회 시 필수 응답 필드**

| 필드 | 설명 | 예시 |
|------|------|------|
| id | 강좌 ID | 1 |
| name | 강좌명 | "자료구조" |
| credits | 학점 | 3 |
| capacity | 정원 | 30 |
| enrolled | 현재 신청 인원 | 25 |
| schedule | 강의 시간 | "월 09:00-10:30" |

> **참고**: 필드명은 자유 (camelCase, snake_case, 한글 모두 허용)

## 제출물 구조

```
project-root/
├── README.md                 ⭐ 필수: 빌드 및 실행 방법
├── CLAUDE.md 또는 AGENTS.md  ⭐ 필수: AI 에이전트 지침
├── (빌드 설정 파일)           ⭐ 필수: build.gradle, package.json, requirements.txt 등
├── docs/
│   ├── REQUIREMENTS.md       ⭐ 필수: 요구사항 분석 및 설계 결정
│   └── (API 문서)            ⭐ 필수: API 명세
├── prompts/                  ⭐ 필수: AI에 입력한 프롬프트 이력
│   └── *.md
└── src/                      ⭐ 필수: 소스 코드
```

### 필수 문서 안내

#### README.md
- 프로젝트 빌드 방법
- 서버 실행 방법
- API 서버 접속 정보 (포트 등)

#### prompts/ (프롬프트 이력)

AI에 입력한 프롬프트를 제출하세요.

**허용되는 형식**:
- 프롬프트를 직접 기록한 마크다운 파일
- AI 도구의 대화 내보내기 (export)
- 자동화된 로깅 (단, 누락되지 않도록 본인이 확인)

**핵심**: AI를 어떻게 활용했는지 평가자가 파악할 수 있어야 합니다. 누락 시 AI 활용 평가에서 감점됩니다.

#### docs/REQUIREMENTS.md
- 기획팀 요청사항에서 도출한 요구사항 분석
- 불명확한 부분에 대한 판단과 결정 사항
- 설계 의사결정 및 근거

#### docs/ 내 API 문서
- 구현한 API의 상세 명세
- 요청/응답 형식
- 각 상황에 대한 응답 정의

### 요구사항 분석

기획팀의 요청사항은 의도적으로 불완전합니다. 실제 업무에서 흔히 마주치는 상황처럼, **명시되지 않은 요구사항을 도출**하고 **합리적인 결정**을 내려야 합니다.

여러분의 분석과 결정은 `docs/REQUIREMENTS.md`에 문서화하세요.

### 명시되지 않은 요구사항

**기획 요청서와 이 문서에 명시되지 않은 모든 사항은 자유롭게 결정하세요.**

예시:
- 인증/인가 (로그인, 권한 관리)
- 수강신청 정책 (취소 기한, 재수강 허용 등)
- 추가 제약사항 (선수과목, 학년별 제한 등)
- 기타 비즈니스 규칙 및 예외 처리

**핵심 기능(강좌 조회, 수강신청/취소, 동시성 제어)**을 우선 구현하고, 나머지는 합리적으로 판단하여 결정하세요.

다음과 같은 **암묵적 결정 사항**도 합리적으로 정하고 문서에 남기세요:
- 시간표 슬롯 정의 (요일/교시/시간 표현)
- 학점 분포 및 강의당 학점 범위
- 시간 충돌 데이터 생성 방식
- 동시성 범위 (동일 학생의 중복/동시 신청 처리 등)

모든 설계 결정은 `docs/REQUIREMENTS.md`에 근거와 함께 기록하세요.

### 평가 안내

평가는 위의 **구현 우선순위**와 동일한 구조로, 비중이 큰 순서대로 나열되어 있습니다.

**1단계: 동작 여부 (가장 큰 비중)**
- 빌드되고 실행 가능한가? 서버가 정상 구동되고 헬스체크에 응답하는가?
- 기본적인 API가 동작하는가?

**2단계: 핵심 기능 (큰 비중)**
- 비즈니스 규칙(정원, 학점, 시간 충돌 등)이 올바르게 동작하는가?
- 동시성 제어가 실제로 견고한가? (동시 요청 시나리오로 직접 검증합니다)

**3단계: 사고의 깊이**
- **AI 활용**: 프롬프트에 담긴 사고 과정과 에이전트 지침의 품질을 중점 평가합니다. 단순히 "만들어줘"가 아닌, 문제를 깊이 이해하고 AI와 협업하는 과정을 봅니다. **이 영역의 비중은 코드 품질보다 큽니다.**
- **설계 문서**: 요구사항 분석의 깊이와 의사결정의 합리성을 평가합니다
- **코드 품질**: 유지보수 가능하고 확장 가능한 코드인지, 어떤 의도로 설계했는지 평가합니다
- **테스트**: 적절한 테스트 커버리지를 갖추고 있는지 평가합니다
- **Git 이력**: 의미 있는 단위로 커밋이 분리되었는지 평가합니다

> 💡 **위 항목이 평가의 전부는 아닙니다.**
>
> 좋은 개발자는 명시적으로 요구받지 않아도, 스스로 필요한 것을 찾아서 합니다.
> 이 과제에도 명시되지 않은 평가 요소가 있으며, 그런 부분을 스스로 고민하고 실천한 후보자에게 추가 점수가 부여됩니다.
